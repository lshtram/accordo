### FAIL — must fix before Phase E

#### FAIL-1 — `recentErrors` ring buffer is never populated
**File:** `packages/browser/src/health-tool.ts`, line 45  
**Severity:** HIGH — functional gap; `browser_health` always returns `recentErrors: []`

```typescript
// Line 45 — allocated but nothing ever pushes into it
const recentErrors: string[] = [];
```

The design spec §3.2 states: *"The relay server already logs events via `onEvent`. The health tool needs a small ring buffer (10 entries) to track recent errors. This buffer lives on the tool builder closure."*

No event subscription API exists on `BrowserRelayLike` — there is no hook from relay error events → `recentErrors.push(...)`. The tests for `recentErrors` only verify `Array.isArray(result.recentErrors)` and `result.recentErrors.length <= MAX_RECENT_ERRORS`, both of which trivially pass for an empty array.

**Fix:** Either (a) add an `onError(cb: (msg: string) => void): void` method to `BrowserRelayLike` and subscribe in `buildHealthTool`, or (b) accept that this phase delivers a non-functional ring buffer with a code comment documenting the intentional deferral (if the team decides the buffer wiring is Phase 2 scope). **A decision is required** — the current state is silently non-functional with no indication in code or tests.

#### FAIL-2 — `isObstructed` does not check descendants (spec deviation)
**File:** `packages/browser-extension/src/content/element-inspector.ts`, line 270  
**Severity:** MEDIUM — behavioral deviation from design spec; causes false positives for `isObstructed: true`

```typescript
// Current — line 270
isObstructed = document.elementFromPoint(centerX, centerY) !== element;
```

Design spec §2.4 states: *"check if returned element is `el` **or a descendant of `el`**"*

The current implementation returns `isObstructed: true` whenever a child element (e.g., an inner `<span>` or `<img>`) is the topmost element at the center point, even though clicking that point still triggers events on the parent `el`. This produces spurious false positives for any element with inline children — a common case.

**Fix:**
```typescript
const top = document.elementFromPoint(centerX, centerY);
isObstructed = top !== null && !element.contains(top);
```

`element.contains(element)` returns `true`, so this correctly handles both "element is topmost" and "a descendant is topmost" as non-obstructed. The test at `element-actionability.test.ts:163` mocks `elementFromPoint` to return `element` directly — this test will still pass, but a new test should be added for the descendant case.

#### WARN-2 — Dead-branch conditional spreads in `buildDetail()`
**File:** `packages/browser-extension/src/content/element-inspector.ts`, lines 289 and 291  
**Severity:** LOW

```typescript
// Line 289 — hasPointerEvents is always boolean, never undefined
...(hasPointerEvents !== undefined ? { hasPointerEvents } : {}),

// Line 291 — clickTargetSize is always { width, height }, never undefined
...(clickTargetSize !== undefined ? { clickTargetSize } : {}),
```

Both guards are always-true; the `{}` branch is dead code. This is misleading — a reader may think these fields can be absent based on some condition.

**Fix:** Use unconditional spreads, or guard with a semantically meaningful condition (e.g., `rect.width > 0 && rect.height > 0` for `clickTargetSize`).

#### INFO-1 — Unnecessary `async` on test functions
**File:** `packages/browser-extension/tests/element-actionability.test.ts`, lines 56, 75, 195  
**Severity:** Informational

Several `it("...", async () => { ... })` callbacks contain no `await` expressions. `async` is unnecessary and may mask future floating-promise issues.

**Fix:** Remove `async` from test callbacks that contain no `await`.

#### INFO-2 — `getDebuggerUrl()` returns a hardcoded placeholder
**File:** `packages/browser/src/relay-server.ts`, line 156  
**Severity:** Informational (deferred by design, but misleading)

Returns `"ws://localhost:9222"` unconditionally. Design spec DEC-022 defers the real implementation. An agent reading this URL would attempt to connect to a potentially wrong address.

**Recommendation:** Return `undefined` (or remove the hardcoded fallback) until the real implementation exists. The `health-tool.ts` already guards on `connected` before including `debuggerUrl` in the response, so returning `undefined` here is safe and less misleading than a hardcoded address.
