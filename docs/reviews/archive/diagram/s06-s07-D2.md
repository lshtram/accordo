## S-07 ‚Äî `cluster.parent` two-pass derivation

### ‚úÖ Two-pass structure is sound

Pass 1: `rawSubgraphs.map(sg => ...)` ‚Äî builds `ParsedCluster[]` without parent.
Pass 2: second `for` loop ‚Äî derives parent by checking whether any cluster's `nodes`
array contains another cluster's ID.

### ‚úÖ `clusterIdSet` construction is correct

```ts
const clusterIdSet = new Set(rawSubgraphs.map(sg => sg.id));
```

Set is built before both passes, ensuring all cluster IDs are available for membership
filtering and parent derivation even if subgraphs are returned in an arbitrary order. ‚úÖ

### ‚úÖ `directMembers` filtering is correct

```ts
const directMembers = sg.nodes.filter(id => !clusterIdSet.has(id));
```

Nodes whose IDs are themselves cluster IDs (i.e. nested sub-clusters) are excluded
from `members`. Direct leaf nodes are preserved. This matches the `ParsedCluster`
interface contract: `members: readonly NodeId[]` ‚Äî "direct members (not members of
nested sub-clusters)". ‚úÖ

### ‚úÖ `nodeToCluster` only populated for direct members

Before S-07, `nodeToCluster.set(nodeId, sg.id)` iterated over all `sg.nodes`
including sub-cluster IDs, which would have incorrectly mapped sub-cluster IDs as
nodes. After S-07 it iterates only over `directMembers`. This is a silent correctness
fix included in S-07. ‚úÖ

### ‚úÖ Parent derivation logic is correct for mermaid 11.x

Mermaid's `getSubGraphs()` for a nested structure like:
```
subgraph outer
  subgraph inner
    A
  end
end
```
returns:
- `{ id: "inner", nodes: ["A"] }`
- `{ id: "outer", nodes: ["inner"] }` (or `["inner", "A"]` depending on version)

The second pass scans each `sg.nodes` for IDs present in `clusterIdSet`. When it
finds `"inner"` in `outer.nodes`, it sets `inner.parent = "outer"`. ‚úÖ

The `!childCluster.parent` guard (only first assignment wins) is the correct
"first parent wins" strategy for ambiguous cases where a cluster appears in multiple
outer clusters' node lists ‚Äî a degenerate case that mermaid should not produce, but
defensive to guard against. ‚úÖ

### ‚öÝÔ∏è Unsafe `as` cast without type guard ‚Äî **minor issue**

```ts
// flowchart.ts:107
(childCluster as { parent?: ClusterId }).parent = sg.id;
```

`ParsedCluster.parent` is declared as `parent?: ClusterId` in `types.ts` ‚Äî it is
already a mutable optional field on the interface. The `as` cast is technically
unnecessary: `childCluster.parent = sg.id` would compile without it, because
`clusters` is typed as `ParsedCluster[]` (not `readonly ParsedCluster[]`).

Per coding guidelines ¬ß1.1: *"No type cast `as X` without narrowing"*. The cast does
not narrow ‚Äî it is used to **assign**, not to narrow a type for consumption. The cast
can be removed entirely. This is a minor style violation.

**Recommended fix:**
```ts
// Before (line 107):
(childCluster as { parent?: ClusterId }).parent = sg.id;
// After:
childCluster.parent = sg.id;
```

### ‚öÝÔ∏è Test gap ‚Äî no test for `cluster.parent` (non-blocking)

The parser test suite (`parser.test.ts` ¬ß6 ‚Äî "Cluster / subgraph extraction") tests
flat (non-nested) subgraphs only. There is no test that:
- passes nested subgraphs to `parseFlowchart`
- asserts `inner.parent === "outer"`
- asserts that `inner` cluster's `members` does not contain sub-cluster IDs

S-07 is a new behavioural change. The absence of a regression test means the logic
is exercised only via the full diagram roundtrip integration test
(`diagram-leaf-integration.test.ts`), which does not use nested subgraphs.

**Flagged as a test-completeness gap; does not block this review given that all
existing tests pass and the logic is correct. A follow-up test should be added.**
