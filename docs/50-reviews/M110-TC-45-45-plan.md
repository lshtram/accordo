#### GAP-A1: Navigate response missing `readyState` (A: 4→5, +1)

**Checklist item:** A2 — "Get load/readiness state: loading | interactive | complete + wait support"

**Current state:** `handleNavigate()` in `control-tool-types.ts` returns `{ success, url, title }`. The `NavigateResponse` interface has no `readyState` field. The relay handler in `relay-control-handlers.ts` uses CDP `Page.getFrameTree` for title but doesn't query `document.readyState`.

**What's missing:** `readyState: "loading" | "interactive" | "complete"` field.

**Fix:**
1. In `relay-control-handlers.ts` (`handleNavigate`): after navigation completes, query `document.readyState` via CDP `Runtime.evaluate` and include in response
2. In `control-tool-types.ts`: add `readyState?: "loading" | "interactive" | "complete"` to `NavigateResponse`
3. Add `waitUntil?: "load" | "domcontentloaded" | "networkidle"` param to `NavigateArgs` (default: `"domcontentloaded"`)

**Also needed for A5:** A4 — iframe enumeration. Currently no explicit frame relationships in any tool output. To reach A=5, consider adding `frameId` to page map nodes when elements are inside iframes, or a `list_frames` action. This is a stretch item — A=5 may be achievable without it if the evaluator considers iframes as "not required by product scope" per the checklist wording.

**Effort:** Small (0.5 day for readyState; +1 day if iframe enumeration is added)  
**Score impact:** A: 4→5  
**Files to change:**
- `packages/browser/src/control-tool-types.ts` — `NavigateResponse`, `NavigateArgs`, `handleNavigate`, `buildNavigateTool`
- `packages/browser-extension/src/relay-control-handlers.ts` — navigate handler implementation

#### GAP-D1: Geometry helpers (D: 2→3, partial)

**Checklist item:** D2 — "Relative geometry helpers (leftOf, above, contains, overlap, distance)"

**Current state:** Bounding boxes exist per-node when `includeBounds: true` on `get_page_map`. No geometry computation exists anywhere in the codebase.

**Strategic options:**
1. **Agent-side computation** — Agents already get bboxes; they could compute `leftOf(a, b)` themselves from bbox coordinates. **Problem:** every agent must implement this, and the checklist explicitly tests for it.
2. **New MCP tool params** — Add query params to `get_page_map` like `relativeToNodeId` that returns geometry relationships. **Problem:** awkward API surface.
3. **New `get_geometry` tool** — A dedicated tool that takes two node IDs and returns spatial relationships. **Problem:** extra round-trip per comparison.
4. **Computed fields on page map nodes** ✅ **RECOMMENDED** — When `includeBounds: true`, compute and include for each node: `viewportIntersection` (0-1 ratio), `containedBy` (parent container nodeId), `zIndex` (stacking order). Plus add a `geometry_helpers` response section with precomputed spatial relationships for interactive elements.

**Fix (recommended approach):**
1. **Content script helper** — New `spatial-helpers.ts` in `packages/browser-extension/src/content/`:
   - `leftOf(a: DOMRect, b: DOMRect): boolean`
   - `above(a: DOMRect, b: DOMRect): boolean`
   - `contains(outer: DOMRect, inner: DOMRect): boolean`
   - `overlap(a: DOMRect, b: DOMRect): number` (0-1 intersection-over-union)
   - `distance(a: DOMRect, b: DOMRect): number` (center-to-center px)
   - `viewportIntersectionRatio(rect: DOMRect, viewport: {width, height, scrollX, scrollY}): number`
2. **Page map enrichment** — When `includeBounds: true`, add to each node:
   - `viewportRatio?: number` (0-1 intersection with viewport) — fixes D4
   - `containerId?: number` (nodeId of nearest semantic container: article, section, aside, dialog, [role=dialog], details) — fixes D5
3. **New MCP action `get_spatial_relations`** — Takes `nodeIds: number[]` and returns pairwise spatial relationships for the given nodes. This satisfies D2 without bloating every page map response.

**Effort:** Large (2.5 days — spatial helpers + page map enrichment + new MCP tool)  
**Score impact:** D: 2→4 (D1 bboxes ✅, D2 geometry helpers ✅, D4 viewport ratios ✅, D5 container grouping ✅)  
**Files to change:**
- NEW: `packages/browser-extension/src/content/spatial-helpers.ts`
- `packages/browser-extension/src/content/page-map-collector.ts` — add viewportRatio, containerId
- `packages/browser/src/page-tool-types.ts` — new `GetSpatialRelationsArgs`, response type
- `packages/browser/src/page-tool-handlers-impl.ts` — new handler

#### GAP-I1: Screenshot redaction + TTL retention (I: 3→5, +2)

**Checklist items:** I1 — "Redaction hooks for PII/secrets in text **and screenshots**" + I4 — "Data-retention controls for snapshots/images"

**Current state:**
- I1-text: ✅ — `redactPII` param on `get_text_map` and `get_semantic_graph` works
- I1-screenshot: ❌ — `redactionWarning` field surfaces the gap but no actual screenshot redaction
- I2: ✅ — Origin policy with allow/deny lists
- I3: ✅ — Audit trail to VS Code output channel
- I4-partial: `RETENTION_SLOTS = 5` FIFO exists, but no TTL-based expiry and no explicit deletion API

**What's needed for I=5:**

**I1 — Screenshot redaction (hard):**
- Option A: **Client-side OCR** — Use Tesseract.js (WASM) to OCR screenshot images, then apply regex redaction patterns to detected text regions and paint over them with solid rectangles. **~300ms latency per image, ~2MB WASM binary.**
- Option B: **Metadata-based redaction** — Cross-reference screenshot bounding boxes with known PII locations from the text map. For elements flagged by text redaction, paint a solid rectangle over their bbox in the screenshot. **No OCR needed, ~50ms, but requires spatial correlation.** ✅ **RECOMMENDED** — achievable without external dependencies, leverages existing infrastructure.
- Option C: **Defer with explicit documentation** — Document that screenshot redaction requires OCR integration and set `redactionWarning` on all screenshot responses. **Current state. Gets I=3 at best.**

**I4 — TTL-based retention:**
1. Add `maxAge?: number` (ms) to `SnapshotRetentionStore` — snapshots older than `maxAge` are evicted on next access
2. Add configurable TTL via VS Code settings: `accordo.browser.snapshotTTL` (default: 300000 = 5 min)
3. Add `browser_clear_snapshots` MCP action (can merge with GAP-G1)

**Fix (recommended — Option B for screenshots):**
1. After capture, if `RedactionPolicy` is active:
   - Query the latest text map snapshot for the same page
   - Find text segments whose `redactionApplied` flag would be true
   - Map their bounding boxes to screenshot coordinates (accounting for scroll offset and device pixel ratio)
   - Draw solid rectangles (matching background color or black) over those regions in the OffscreenCanvas
2. Add `screenshotRedactionApplied?: boolean` to `CaptureRegionResponse`
3. Fall back to `redactionWarning` if no text map snapshot is available for cross-referencing

**Effort:** Medium-Large (2 days for screenshot redaction Option B; 0.5 day for TTL)  
**Score impact:** I: 3→5  
**Files to change:**
- `packages/browser-extension/src/relay-capture-handler.ts` — bbox-based screenshot redaction

- `packages/browser/src/snapshot-retention.ts` — TTL-based eviction
- `packages/browser/src/page-tool-types.ts` — `CaptureRegionResponse.screenshotRedactionApplied`

## 7. What's Hard and Why

| **GAP-D2: Z-order** | Hard | `elementFromPoint` is synchronous but O(n) per sampled point. Stacking context computation requires walking the ancestor chain. False positives from `position: fixed` elements (headers, footers). |
| **GAP-I1: Screenshot redaction** | Hard | Spatial correlation between text bboxes and screenshot pixels requires accounting for DPR, scroll offset, and CSS transforms. No OCR fallback means we miss PII that's only in images (logos, embedded text in graphics). |

| **GAP-F1: Eventability** | Medium | `elementFromPoint` for obstruction detection + `pointerEvents` check is straightforward. Touch target sizing requires understanding CSS box model interactions. |
